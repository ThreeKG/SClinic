# Final Proposal Edits & Recommendations

## What the DOCX adds that the plan should reflect

The proposal is broader and cleaner than the current phase-2 planning. It introduces:

- Phase 1: launch stabilisation before the redesign
- Phase 2: brand refresh + website redesign + digital patient experience
- Core vs optional scope separation
- Standard integrations only where APIs/connectors already exist
- Advanced modules to be discovered, scoped, and priced separately
- Commercial notes: phase 1 and phase 2 should be priced independently
- Timeline assumptions: access, content readiness, feedback cycles, approvals, and integration availability all affect delivery

## What’s missing / needs improvement in the final proposal

| Area | Status | Recommendation |
|---|---|---|
| Phase 1 scope | Missing from the current phase-2 plan | Add a short launch-stabilisation phase before the redesign scope |
| Scope boundaries | Some overlap between core and optional items | Make a clean “core build vs optional modules” split |
| Integrations | Mentioned, but too vague | Name what counts as standard integration and what is treated as custom/advanced |
| Commercial structure | Not explicit in the plan | State that phases are priced independently |
| Timeline assumptions | Not fully spelled out | Add dependency notes for access, content, approvals, and 3rd-party systems |
| Acceptance criteria | High-level only | Add launch/approval criteria for each phase |
| Platform decision | Open question, but not gated | Make CMS/platform choice a formal decision point before build starts |
| Patient journey features | Broadly referenced | Define what is core in UX vs what becomes a separate module |
| Arabic / multilingual | Good as optional, but should be explicit | Keep it as future scope unless the client confirms it now |

## My revision recommendation

### 1) Add a Phase 1 section

Use the proposal’s launch-stabilisation idea explicitly:

- access review
- urgent fixes
- form checks
- content cleanup
- mobile/layout fixes
- SEO basics
- speed improvements
- tracking readiness
- QA + launch support

### 2) Keep Phase 2 focused on the redesign

Core Phase 2 should be:

- brand identity refresh
- homepage and internal page redesign
- design system
- content presentation improvements
- platform direction
- standard integrations only

### 3) Move advanced items out of core scope

These should be explicitly separate:

- CRM
- booking dashboard
- automated patient communication
- AI enquiry filtering
- advanced booking logic
- Arabic expansion
- e-commerce
- campaign-specific extras

### 4) Add a “proposal guardrails” section

Include:

- what is included
- what is excluded
- what depends on third-party systems
- what needs client approval before build

## Bottom line

The final proposal is strong conceptually, but it would be better if it were made:

- more explicit about Phase 1
- tighter about scope
- clearer about integrations
- more commercial / delivery-ready
